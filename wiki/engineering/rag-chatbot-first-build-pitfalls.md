# 首次构建 RAG 问答 Chatbot 的工程踩坑

**日期**：2026-06-27
**触发事件**：首次为内部 Slack 频道构建三层降级 RAG 问答 bot，上生产前做完整 code review，发现 3 个会阻断上线的 Critical 隐患
**适用范围**：任何"机器人轮询消息源 → RAG 检索 → LLM 生成 → 回帖"的问答 bot（Slack/Telegram/Discord/IM 均通用），尤其首次做的人

---

## 核心原则

> **聊天机器人是一个会"对自己说话"的系统。**最隐蔽的生产事故不是答错，而是 bot 把自己的回复当成新问题 → 无限自回复风暴。
> 其次：缓存命中绕过了所有防编造逻辑；RAG bot 的"防编造/降级/去重"全是分支逻辑，没有测试就是裸奔。

---

## 坑 1（最隐蔽，最高危）：共享账号下的自回复死循环

### 症状
bot 回复一条消息后，下一轮轮询又把这条回复读成新消息、再次回答，循环爆炸；或在某次"让回复带上触发词前缀"的无害改动后突然炸开。

### 根因
很多 IM 集成里，**bot 和人类用户共用同一个账号 ID**（例如机器人用某员工的 OAuth token 发消息）。此时：

```
bot 自己发的回复 → 不带 bot_id 字段（因为是用「用户身份」发的）→ 看起来就是一条普通用户消息
```

如果轮询循环用 `if msg.get("bot_id"): skip` 来过滤自己的消息，**对共享账号完全失效**——bot 的回复带不带 `bot_id` 取决于发送通道，用用户 token 发的就是空。

更危险的是：系统当下不死循环，往往**纯属偶然**——只因为"回复正文恰好不以触发词开头"。任何让回复以触发词/项目名/回显用户问题开头的改动，会瞬间引爆。

### 解法：用「发送方追加的稳定签名」判定自己的消息，而非账号 ID
共享账号下，**唯一可靠的"这是我发的"标记，是发送 API 自动追加的签名/footer**（如某些 OAuth client 发消息会带 `*Sent using* ...` 尾签）。

```python
SENT_SIGNATURE = "*Sent using"   # 你的发送通道追加的稳定标记

def is_own_message(m):
    if m.get("bot_id"):                       # 走 bot token 时有效
        return True
    return SENT_SIGNATURE in (m.get("text") or "")   # 共享账号唯一可靠判据

# 在「顶层消息」和「线程内回复」两个过滤点都要用，别漏其一
```

**不要**只用 `user == BOT_USER_ID` 判定——因为人类也共用这个 ID，会把真人提问误过滤掉。签名才是区分"机器输出 vs 人类输入"的那把钥匙。

### 上线前铁律
- 在两个过滤点（顶层消息扫描 + 线程回复扫描）都应用 `is_own_message`，常见漏改是只改了一处。
- **必须用真实账号在测试频道实跑一轮**，确认 bot 不会回应自己的上一条回复。单测能覆盖逻辑（含"回复以触发词开头"的最坏情况），但端到端实测验证签名真的被追加。

## 坑 2：缓存命中绕过了防编造逻辑（丢来源 + 永不过期）

### 症状
- 同一个问题第二次问，秒回但**答案不带来源**（违反"每个论断必带出处"）。
- 数据每日刷新后，旧答案仍被缓存命中，**陈旧答案当新鲜答案返回**。

### 根因
语义/精确缓存通常只存 `{question: answer}`，命中时直接返回 answer，**绕过了正常应答路径里挂载 sources、做新鲜度校验的所有逻辑**。缓存成了防编造体系的后门。

### 解法
1. **缓存里一并存 sources**，命中时回填——保证缓存答案和实时答案一样带源。
2. **加 TTL，与数据刷新周期对齐**（数据每日刷新 → 缓存 24h 过期）。缓存答案的存活期绝不能超过它所依据的数据。
3. 时间戳解析失败/缺失 → 一律当过期处理（fail-safe）。

```python
CACHE_TTL_SECONDS = 24 * 3600   # 对齐每日刷新

def cache_get(q):
    hit = store.get(norm_key(q))
    if not hit:
        return None
    try:
        age = time.time() - parse_ts(hit["ts"])
    except Exception:
        return None                 # 时间戳坏 → 视为过期
    return None if age > CACHE_TTL_SECONDS else hit

def cache_put(q, answer, tier, sources=None):
    store[norm_key(q)] = {"answer": answer, "tier": tier,
                          "sources": sources or [], "ts": now_str()}
```

> 通用教训：**任何"快路径/缓存/早返回"分支，都要自问——它是否绕过了慢路径里的安全/合规逻辑（来源、新鲜度、护栏）？** 绕过了就要在快路径里补回。

## 坑 3：RAG bot 的核心防护全是分支逻辑，零测试 = 裸奔

### 症状
代码工艺很好（防编造 prompt、三层降级、去重），但全仓 0 个测试。任何改动都可能悄悄引入自回复循环、错误降级、或破坏防编造，且无人察觉。

### 根因
RAG/chatbot 的"质量"不在某个算法，而在一堆**判定分支**：触发词解析、自己消息过滤、"答案是否信息不足要降级"、路由分类。这些都是纯逻辑，**恰恰最适合单测、也最容易在迭代中被改坏**。

### 解法：写"不依赖 LLM/向量库/网络"的纯逻辑单测
把可独立测试的判定逻辑抽成纯函数，用真实样本测，CI/pre-commit 秒级跑完：

```
□ 触发词解析    parse_trigger：全/半角分隔符、前导空格、不在句首、空输入
□ 自回复过滤    is_own_message：带签名=自己 / 共享账号的人类提问≠自己 / 最坏情况
□ 缓存          带 sources 回填 + TTL 过期 + 坏时间戳当过期
□ 路由分类      聚合类 / 显式全局检索 / 项目问答
```

关键：**用 monkeypatch 把缓存路径、state 文件路径指到临时目录**，测试零副作用、不碰真数据、不调 LLM。一套 26 个测试 <2s 就能给整个判定层兜底。

## 坑 4：机器人的 state 文件必须原子写

### 症状
长跑轮询进程崩溃/被 kill 时，正在写的游标文件（`cursor.json`/已处理线程表）被截断成半个 JSON，重启后解析失败 → 游标重置 → 重复回答历史消息，或自回复去重失效。

### 根因
`json.dump(obj, open(path, "w"))` 不是原子的——写到一半进程死掉就留下损坏文件。游标/去重表是机器人的"持久化锚点"，损坏直接破坏崩溃重试和自回复去重语义。

### 解法：temp 文件 + os.replace（同目录 rename 是原子的）
```python
def atomic_write_json(path, obj):
    os.makedirs(os.path.dirname(path), exist_ok=True)
    fd, tmp = tempfile.mkstemp(dir=os.path.dirname(path), suffix=".tmp")
    try:
        with os.fdopen(fd, "w", encoding="utf-8") as f:
            json.dump(obj, f, ensure_ascii=False)
        os.replace(tmp, path)        # 原子替换，崩溃也不会留半截文件
    except Exception:
        try: os.unlink(tmp)
        except OSError: pass
        raise
```

## 上线前 Pre-flight 检查清单

```
□ 自回复过滤用「发送方签名」而非账号 ID，且顶层 + 线程两处都应用
□ 真实账号实测一轮：bot 不回应自己的回复
□ 缓存命中带 sources，且有 TTL（对齐数据刷新周期）
□ 任何快路径/早返回，确认没绕过防编造/护栏/新鲜度逻辑
□ 判定层有纯逻辑单测（不依赖 LLM/向量库/网络），CI 秒级跑
□ 游标/去重表等 state 文件原子写
□ 异常信息不直接回贴用户（可能泄漏内部路径/堆栈），详情仅进日志
□ 游标推进顺序：回帖成功后才推进，宁可重答不丢问
```

## 通用工程箴言（脱离任何具体平台都成立）

1. **聊天机器人会对自己说话**——先想清楚"如何 100% 可靠地识别自己发的消息"，再写回帖逻辑。
2. **缓存是安全逻辑的后门**——任何绕过慢路径的快路径，都要补回慢路径的安全保证。
3. **chatbot 的质量住在分支里**——分支逻辑必须有纯函数单测兜底。
4. **长跑进程的 state 必须原子写**——崩溃随时会来。
5. **代码工艺好 ≠ 可上生产**——独立 reviewer（零共享上下文、fail-closed）总能发现作者自己看不见的盲点。
