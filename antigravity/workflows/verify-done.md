# Workflow: /verify-done

> 完成前验证 Workflow，由 `general-global-rule.md §5.3` 强制触发。
> 本文件只定义**验证动作序列**，所有规则依据见 `general-global-rule.md` 对应章节。
> 最后更新:2026-04-22

---

## 触发条件
任何任务宣称"完成"之前的**最后一个动作**必须是本 Workflow。
- 未调用本 Workflow 就说"完成"视为违规（依照 §5.4）
- 违规会被要求回炉重做并记录为 lesson

## 核心原则
- **任何一步失败都不得跳过**，必须修复后重跑该步
- **没有测试不等于跳过验证**——手动验证照样要做
- 所有验证结果必须输出给用户，不能只在内部判断

## 第 0 步：核对 TODO 清单

打开 `tasks/todo.md`，找到本次任务的 TODO 段落，**逐项核对**：

1. 每个 `[ ]` 步骤是否已实际完成
2. 有没有**隐含但漏做**的步骤（例如"新增函数"但忘了"更新调用方"）
3. 有没有与计划**不一致的意外改动**（例如计划说改 2 个文件，实际改了 5 个）

**输出格式**：
TODO 核对

 步骤 1：<完成情况简述>
 步骤 2：<完成情况简述>
 步骤 3：<未完成，原因>
异常改动：<有/无，如有列出>


如有 `[ ]` 未打钩或异常改动，**停下**汇报用户，不得进入下一步。

---

## 第 0.5 步：TDD 前置检查（核心业务逻辑改动必做）

依照 `general-global-rule.md §2.6` 的 Test-First 原则，在跑 lint / 测试之前，先确认本次改动是否**应该**有测试。

### 0.5.1 判断改动类型

根据 `tasks/todo.md` 里"实际改动文件"清单，对照下表：

| 改动涉及 | 是否必须 TDD | 处理方式 |
|---|---|---|
| LLM fallback / 调度逻辑 | ✅ 必须 | 进入 0.5.2 |
| 爬虫管道、数据清洗核心流程 | ✅ 必须 | 进入 0.5.2 |
| Pydantic Schema 修改影响线上数据 | ✅ 必须 | 进入 0.5.2 |
| 关键 bug 修复（业务流程中断、数据丢失） | ✅ 必须 | 进入 0.5.2 |
| 配置项调整、文档、UI 微调 | ❌ 豁免 | 跳过本步骤，明确声明"豁免理由：<类型>" |
| Trivial 档任务 | ❌ 豁免 | 跳过本步骤，明确声明"Trivial 档豁免" |

### 0.5.2 检查测试是否先于业务代码存在

执行：

```bash
ls tests/ 2>/dev/null
```

**判断分支**：

- **`tests/` 目录不存在** → ⚠️ **停下汇报用户**：
  > "本次改动属于核心业务逻辑（<具体类型>），但项目无 tests/ 目录，违反 §2.6 Test-First 原则。是否：(a) 现在补 tests/ 目录和测试用例 (b) 用户确认豁免本次 TDD（需说明理由）"
  
- **`tests/` 目录存在但本次改动无对应测试** → ⚠️ **停下汇报用户**：
  > "改动涉及 <文件名>，但未发现对应的测试用例（搜索 tests/ 下没有 test_<模块名>.py）。是否：(a) 现在补测试 (b) 用户确认豁免"

- **`tests/` 目录存在且有对应测试** → ✅ 进入 0.5.3

### 0.5.3 验证测试与代码的时间顺序

执行（针对每个核心改动文件）：

```bash
git log --diff-filter=A --pretty=format:"%h %ad %s" --date=short -- tests/test_<模块名>.py | head -1
git log --diff-filter=AM --pretty=format:"%h %ad %s" --date=short -- <文件名> | head -1
```

**理想情况**：测试文件的首次提交时间 ≤ 业务代码本次改动时间。

**容忍情况**：测试文件较新但**先于本次业务改动 commit**（比如同一次 commit 里 test 写在前 N 行、code 写在后 N 行）。

**违反情况**：业务代码已经写完才补测试 → 警告用户"本次未严格走 TDD 顺序，建议下次先写 failing test"，但不阻止流程继续。

### 0.5.4 反模式

- ❌ 跳过 TDD 检查只因为"我看代码没问题"
- ❌ 用一个 `assert True` 的空测试糊弄
- ❌ 在 0.5.2 报警后用户不答应豁免却继续往下走

## 第 1 步：静态检查（Ruff）

### 1.1 Lint 检查
执行命令：
```bash
ruff check .
```

**期望输出**: `All checks passed!`

**失败处理**：
- 自动修复可修项：`ruff check --fix .`
- 无法自动修复的警告：逐项分析是否需要修改代码，或在 `pyproject.toml` / `.ruff.toml` 中添加合理的 ignore 规则（需用户确认）
- **严禁**用 `# noqa` 注释批量掩盖——每个 `noqa` 必须附中文注释说明为什么必须忽略

### 1.2 格式检查
执行命令：
```bash
ruff format --check .
```

**期望输出**: 无 "Would reformat" 提示

**失败处理**：
- 自动格式化：`ruff format .`
- 重新运行 `--check` 确认

## 第 2 步：测试（有则必跑）

### 2.1 判断项目是否有测试
执行：
```bash
ls tests/ 2>/dev/null && echo "HAS_TESTS" || echo "NO_TESTS"
```

### 2.2a 如果有 `tests/` 目录
执行：
```bash
pytest -x --tb=short
```

- `-x`：遇到第一个失败即停
- `--tb=short`：简短回溯，避免输出刷屏

**失败处理**：修复失败的测试，重跑，直到全部通过。

### 2.2b 如果没有 `tests/` 目录
**明确声明**："本项目暂无测试目录，跳过 pytest。完全依赖第 3 步的场景化手动验证。"

**不要**因为"没测试"就降低验证严格度——接下来的手动验证要更认真。

---

## 第 3 步：场景化手动验证（核心，永远必做）

根据本次改动涉及的模块，**对号入座**执行对应场景的验证。多项适用时全部执行。

### 3.1 如果改动涉及 **Pydantic 模型**
- 构造 **3 种输入**：合法边界值、非法类型、缺失必填字段
- 对每种输入跑一次模型实例化，确认 Pydantic 的校验行为符合预期
- 输出：三种输入的实际 ValidationError 信息（或通过结果）

### 3.2 如果改动涉及 **LLM 调用链路**（依照 §3）
- 确认模型 ID **不是硬编码**（grep 搜索代码里有没有 `gemini-1.5-pro`、`claude-` 等字面量）
- 确认级联 fallback **仍然生效**——具体验证：
  - 一级模型故意传错 API Key，应自动降到二级
  - 二级也失败时，应自动切平台（Vertex → AI Studio）
  - 全部失败时，异常信息包含"所有候选链路已尝试"字样
- 输出：验证结果摘要

### 3.3 如果改动涉及 **爬虫/数据清洗**（youtube_transcript_api / trafilatura / 微信等）
- 用一个**真实样本 URL** 跑一次完整流程
- 确认：
  - 反爬机制仍然有效（例如 UA 切换、重试逻辑）
  - 输出数据结构符合预期（字段齐全、编码正确）
- 输出：样本 URL + 输出前 100 字符

### 3.4 如果改动涉及 **FastAPI 端点**
- 启动 `uvicorn` 本地服务
- 用 `curl` 或 `httpie` 打一次请求：
```bash
  curl -X POST http://127.0.0.1:8000/<endpoint> \
       -H "Content-Type: application/json" \
       -d '<最小 payload>'
```
- 确认：
  - HTTP 状态码符合预期
  - 响应 JSON 结构正确
  - 错误输入能触发正确的 4xx 响应（不是 500）
- 输出：请求命令 + 响应摘要

### 3.5 如果改动涉及 **前端（HTML/CSS/Vanilla JS）**
- 本地打开页面，**手动点一遍**所有受影响的交互
- 打开浏览器 DevTools 的 Console 标签
- 确认：
  - 无 JavaScript 报错
  - Fetch 请求能正常发出并拿到响应
  - UI 渲染符合预期
- 输出：点击路径 + Console 截图/摘要

### 3.6 如果改动涉及 **config.py 或环境变量**
- 确认改动后的配置不会导致**启动失败**
- 缺失的环境变量要有**清晰的错误提示**（不是裸 `KeyError`）
- 输出：启动一次验证的日志尾部

## 第 4 步：自问"高级工程师会批准吗"

依照 `general-global-rule.md §4.1`，在宣称完成前暂停一次，**严肃回答**：

1. **命名**: 新增的函数、类、变量名称是否表意清晰？有没有 `data2`、`tmp`、`handle` 这种偷懒命名？
2. **职责单一**: 新增的函数是不是只做一件事？有没有 50 行以上的巨型函数？
3. **异常处理**: 有没有裸 `except:` 或 `except Exception: pass` 吞异常？
4. **魔法数字**: 代码里有没有未经解释的 `42`、`300`、`0.75` 之类的硬编码？
5. **副作用**: 新增代码有没有修改全局状态、打印日志、写文件等隐含副作用？有的话是否必要？
6. **向后兼容**: 改动会不会破坏已有的调用方？

**输出格式**：
高级工程师视角审查

命名: ✅ / ⚠️ <若有问题具体说明>
职责单一: ✅ / ⚠️
异常处理: ✅ / ⚠️
魔法数字: ✅ / ⚠️
副作用: ✅ / ⚠️
向后兼容: ✅ / ⚠️


任何一项 ⚠️ 都必须在提交前修掉或在回复中明确声明"用户确认保留"。

**只有**完成上述 6 个步骤才能说"完成"。中途任何失败或异常都要停下汇报。

## 第 7 步：验证失败的自治反思路径（Reflexion Loop）

依照 `general-global-rule.md §4.10` 的自治反思内循环规则。

### 7.1 触发条件

第 1-4 步任意一步**失败**时，**不要立刻停下问用户**，而是按以下流程：

| 失败步骤 | 是否触发 Reflexion |
|---|---|
| 第 1 步 Ruff lint/format | ✅ 自动修复后触发（`ruff check --fix .` + `ruff format .`） |
| 第 2 步 Pytest | ✅ 触发 Reflexion Loop |
| 第 3 步 场景化验证 | ✅ 触发 Reflexion Loop |
| 第 4 步 高级工程师审查 | ⚠️ 仅 ⚠️ 项触发（✅ 项不触发） |

### 7.2 Reflexion Loop 执行

调用 `/self-correct` Workflow（详见 `workflows/self-correct.md`），传入：

- 失败的具体步骤
- 报错信息全文
- 当前 todo.md 的任务上下文

`/self-correct` 会按 MAX_ITERATIONS=3 自动尝试修复，并在退出时返回三种状态之一：

- **SUCCESS**：修复成功，回到第 1 步重新跑 verify-done 全流程
- **DRIFT_DETECTED**：触发 anti-drift 短路，停下汇报用户
- **MAX_REACHED**：3 次都失败，停下汇报用户并附上完整尝试历史

### 7.3 不可触发自治重试的例外

依照 `general-global-rule.md §4.10` 末尾的例外条款：

- ❌ 涉及 `git push` / `git reset --hard` / `rm` / 删除数据库记录等不可逆动作
- ❌ 涉及 LLM API 真实付费调用（如 fallback 验证已经触发计费）
- ❌ 涉及外部系统副作用（发邮件、推送通知、写线上数据库）

以上情况**必须立即停下**，向用户报告：「检测到不可逆动作失败，禁止自治重试，需要您介入」。

### 7.4 Reflexion 失败后的回滚选项

如果 `/self-correct` 返回 MAX_REACHED 或 DRIFT_DETECTED，提示用户两个选项：

1. **手动接手**：用户阅读尝试历史后亲自修复
2. **回滚到检查点**：调用 `/rollback`（详见 `workflows/rollback.md`），回到 `/plan-task` 第 0.5 步建立的 Git checkpoint

输出格式：

```
🚨 Verify 失败且 Reflexion Loop 已用尽

失败步骤：<step>
最终错误：<error>
尝试历史：<3 次的 reflection + 修改差异>

请选择：
(a) 我要手动接手修复
(b) 回滚到 checkpoint <hash>，重新规划
(c) 其他方案（请说明）
```

## 反模式（禁止行为）

- ❌ 跳过 `ruff check`（即使觉得"改动很小"）
- ❌ 因为"没装 pytest / 没测试"就跳过场景化手动验证
- ❌ 场景化验证敷衍（只说"已验证"不给证据）
- ❌ 把 Ruff 警告用 `# noqa` 批量掩盖
- ❌ 自问优雅只打勾不真审
- ❌ 未完成全部 6 步就说"完成"
- ❌ TODO 清单不归档，留一堆未勾选项

---

## 与其他 Workflow 的关系

- 本 Workflow 由 `/plan-task` 生成的 TODO 清单**闭环**——plan 的最后一项永远是 `验证：调用 /verify-done`
- 验证失败发现需要大改 → 重新调用 `/plan-task` 规划修复方案
