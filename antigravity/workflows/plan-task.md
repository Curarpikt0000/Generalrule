# Workflow: /plan-task

> 规划节点 Workflow，由 `general-global-rule.md §5.1` 强制触发。
> 本文件只定义**动作序列**，所有规则依据见 `general-global-rule.md` 对应章节。
> 最后更新：2026-04-22

---

## 触发条件
任何编码任务收到后的**第一个动作**必须是本 Workflow。无一例外。

## 输出承诺
走完本 Workflow 后，agent 必须产出：
1. 一份写入 `tasks/todo.md` 的可勾选清单
2. 一段中文自然语言的**计划摘要**发给用户
3. 明确问句："是否确认按此计划执行？"并**停下等待用户确认**

**未收到用户明确"确认"字样前，严禁动任何代码。**

---

## 第 0 步：判断任务复杂度

快速判断当前任务属于哪一档，决定走轻量模式还是完整模式：

| 档位 | 判定标准 | 走哪个模式 |
|---|---|---|
| **Trivial** | 1-2 步即可完成，无架构影响（改文案、改一行配置、加一条日志） | **轻量模式** |
| **Normal** | 3+ 步骤，涉及多个文件，或需要新增函数/类 | **完整模式** |
| **Complex** | 涉及架构决策、新增模块、修改核心链路（LLM 调度、爬虫管道） | **完整模式 + 自问优雅** |

> **模糊时一律升档**——宁可走完整流程多花 30 秒，不要漏掉影响架构的决策。

---

## 第 0.5 步：Git 安全快照（Normal/Complex 必做，Trivial 可跳过）

依照 `general-global-rule.md §5.6` 的"Snapshot & Rollback"规则，在写任何代码前必须建立 Git 检查点。

### 0.5.1 检查工作区状态

执行：

```bash
git status --short
```

**判断分支**：

| 工作区状态 | 处理方式 |
|---|---|
| 干净（无输出） | ✅ 直接进入 0.5.2 |
| 有未提交改动 | ⚠️ **停下询问用户**：「检测到未提交改动：<改动列表>。是否：(a) 先 commit (b) 先 stash (c) 直接覆盖（不推荐）？」 |
| 不在 Git 仓库内 | ⚠️ **停下询问用户**：「当前目录不是 Git 仓库，本次任务无回滚保护。是否仍要继续？」 |

**不要**在工作区不干净时直接动手——一旦失败无法干净回滚。

### 0.5.2 创建 Auto-checkpoint

工作区干净后，执行：

```bash
git commit --allow-empty -m "Auto-checkpoint before <task-name>"
```

记录返回的 commit hash，写入 `tasks/todo.md` 的"任务"段落顶部：

```markdown
**Checkpoint**: <short-hash> (auto-created at YYYY-MM-DD HH:MM)
```

### 0.5.3 回滚指引

在计划摘要的末尾，提示用户：

> 💾 已建立 Git 检查点 `<short-hash>`。如本次任务执行失败且 `/self-correct` 也无法修复，可调用 `/rollback` 或手动执行 `git reset --hard <hash>` 回到任务开始前。

### 0.5.4 例外说明

**Trivial 档**可跳过本步骤（改动小，回滚成本不如直接重新改），但必须在计划摘要中明确声明"Trivial 档跳过 Git checkpoint"。

**纯文档/配置改动**（无业务逻辑）也可跳过，但同样要明确声明。

### 0.5.5 反模式

- ❌ 工作区有未提交改动时直接 commit 进 checkpoint（会污染历史）
- ❌ checkpoint 后忘记把 hash 写入 todo.md（失败后找不到回滚点）
- ❌ 静默跳过此步骤（必须明确声明跳过及原因）

## 第 1 步：读取历史经验（强制）

**无论轻量/完整都必须执行。**

1. 读取当前项目的 `tasks/lessons.md` 全文
2. 提取所有 lesson 的**关键词**和**规则**字段
3. 对照当前任务，识别可能适用的 lesson
4. 在计划摘要中明确列出："本次将遵守以下 lesson：L-xxx, L-yyy"；如无匹配则写"无历史经验匹配"

> 如果 `tasks/lessons.md` 不存在或为空，跳过此步但在摘要中声明"项目暂无历史 lessons"。

---

## 第 2 步：读取项目级规则（强制）

读取当前项目根目录的 `AGENTS.md`，识别本项目特有约束（禁用库、架构红线、业务术语）。在计划中明确哪些约束会影响本次任务。

如果 `AGENTS.md` 不存在，声明"项目无特殊约束"。

---

## 第 2.5 步：查询 Wiki 领域知识（Normal/Complex 必做，Trivial 可跳过）

根据任务关键词，到共享 Wiki 查询相关领域的已有知识：

1. **读取索引**：先读 `/Users/chaojin/Antigravity Projects/Generalrule/wiki/index.md`
2. **定位领域**：根据任务内容匹配领域（llm / frontend / engineering / crawler / image-gen）
3. **读取相关页面**：进入对应领域目录，读取与任务相关的 Wiki 页面
4. **应用知识**：在后续规划中明确列出"本次参考的 Wiki 知识：[[页面名]]"

**输出要求**：
- 列出读取了哪些 Wiki 页面（或声明"Wiki 对应领域为空/无相关页面"）
- 说明 Wiki 知识如何影响本次方案选择

**如果 Wiki 对应领域为空**：继续执行，但在计划摘要中提示"本次经验将有可能成为该领域的首条知识"。

---

## 第 3 步：搜索现成方案（Normal/Complex 必做，Trivial 可跳过）

依照 `general-global-rule.md §7` 的"开发前必搜原则"，在动手写新代码前：

1. 调用 `@find-skills <需求关键词>` 检索 Skills 生态
2. 若无匹配，用 web_search 查 PyPI / GitHub 上的开源库
3. 评估标准：
   - Star 数（>500 优先）
   - 最近 commit（6 个月内有维护）
   - 文档完整度
   - License 是否兼容商用（如果项目涉及商用）

在计划中明确写出：
- 找到的候选方案（至少列 2-3 个，或说明"未找到合适方案，将手写"）
- 选择哪个 + 选择理由

> 本步骤的详细 SOP 见 `find-skill-first.md` Workflow，必要时可调用 `/find-skill-first`。

---

## 第 4 步：写出 TODO 清单

将任务拆分为可勾选的子步骤，**写入** `tasks/todo.md`。格式：

```markdown
## 任务：<任务标题> (YYYY-MM-DD HH:MM)

### 背景
<1-2 句说明任务来源和目标>

### 参考 Lessons
- L-xxx：<如何影响本次执行>
- L-yyy：<如何影响本次执行>

### 选择方案
- 候选：<方案 A / B / C>
- 采用：<最终方案> + 理由

### TODO
- [ ] 步骤 1：<具体动作>
- [ ] 步骤 2：<具体动作>
- [ ] 步骤 3：<具体动作>
- [ ] 验证：调用 `/verify-done`

### 预计影响文件
- `path/to/file1.py`：<改什么>
- `path/to/file2.py`：<改什么>

### 风险/注意
- <可能的副作用、回退方案、需要用户确认的歧义点>
```

> TODO 清单**追加**到文件末尾，不覆盖历史任务，便于回溯。

---

## 第 5 步：非平凡改动的"自问优雅"（仅 Complex 档）

依照 `general-global-rule.md §4.1`：

在汇报前，暂停一次并在内心（或输出中）回答：

- "有没有更优雅的方案？"
- "这个方案是不是只是在打补丁？"
- "高级工程师会批准这个吗？"

如果想到了更优雅的方案，把**原方案 vs 优雅方案**两种一起写进计划摘要，让用户选。

---

## 第 6 步：汇报并等待确认

输出格式（发给用户）：📋 任务计划
任务档位: <Trivial/Normal/Complex>
参考 Lessons: L-xxx, L-yyy（或"无匹配"）
项目约束: <相关约束> 或 "无"
方案选择: <方案名> —— <一句话理由>
预计影响文件: <N 个>: a.py, b.py, c.py
主要步骤:

...
...
...

风险点: <如有>
TODO 清单已写入 tasks/todo.md。
❓ 是否确认按此计划执行？

**然后停下。不写代码。不执行。等用户回复"确认"、"OK"、"go"等明确同意词。**

如果用户提出修改意见，**回到对应步骤重新规划**，不要直接跳过确认流程。

---

## 反模式（禁止行为）

- ❌ 跳过读 lessons.md，直接规划
- ❌ 不写 todo.md，只在对话里口头计划
- ❌ 规划完不等确认就开写
- ❌ Complex 任务跳过"自问优雅"步骤
- ❌ 搜索步骤敷衍（只说"搜了一下没找到"不给候选）
- ❌ 把 Normal 档误判为 Trivial 跳过搜索

---

## 与其他 Workflow 的关系

- 规划完成并获得用户确认 → 开始执行（无专属 Workflow，依照 `general-global-rule.md §4` 的工作习惯执行）
- 执行过程中需要搜索新方案 → 调用 `/find-skill-first`
- 宣称完成前 → 调用 `/verify-done`
