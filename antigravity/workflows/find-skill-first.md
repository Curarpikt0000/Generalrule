# Workflow: /find-skill-first

> Skill 优先搜索 Workflow，由 `general-global-rule.md §5.2` 和 §7 强制触发。
> 本文件只定义**搜索动作序列**，所有规则依据见 `general-global-rule.md` 对应章节。
> 最后更新：2026-04-22

---

## 触发条件
**任何新功能开发前必须调用**。无论功能看起来多简单——"先搜再决定"是红线。

具体触发场景：
- 计划要写一个新函数/类/模块
- 计划要调用一个此前项目没用过的外部 API
- 计划要实现一个"看起来常见"的功能（爬虫、清洗、解析、转换、格式化等）
- 计划要写超过 30 行的新代码

## 输出承诺
走完本 Workflow 后，agent 必须产出：
1. 一份**候选方案对比表**（至少 2-3 个候选 或 明确声明"三层漏斗均未找到"）
2. 推荐方案 + 选择理由
3. **用户确认**后才能进入编码

---

## 第 0 步：需求抽象

在搜索之前，先把用户的口头需求**抽象成关键词集合**。

### 0.1 需求拆解
把需求拆成 3 个层次：
- **核心动作**：这个功能本质在做什么？（抓取 / 解析 / 转换 / 验证 / 调度 / ...）
- **领域对象**：作用于什么类型的数据？（PDF / YouTube 字幕 / 微信文章 / JSON schema / ...）
- **特殊约束**：有什么特殊要求？（反爬 / 流式 / 异步 / 缓存 / ...）

### 0.2 生成关键词矩阵
至少生成 3 组**不同角度**的搜索关键词，用于后续 3 个渠道：
组 1（英文通用术语）: "pdf text extraction python"
组 2（具体库名猜测）: "pdfplumber pypdf2 pdfminer"
组 3（场景化描述）: "scanned pdf ocr readable"

### 0.3 输出需求档案
需求档案

核心动作: <动词>
领域对象: <名词>
特殊约束: <如有>
搜索关键词: [k1, k2, k3]


---

## 第 1 步：第一层漏斗——Skills 生态搜索（最高优先级）

### 1.1 调用 find-skills
使用已安装的 `find-skills` 工具，对每组关键词各搜一次：
@find-skills <关键词组 1>
@find-skills <关键词组 2>
@find-skills <关键词组 3>

### 1.2 评估搜索结果
对每个返回的 Skill，按以下维度打分（1-5 分）：

| 维度 | 说明 | 最低接受分 |
|---|---|---|
| **覆盖度** | Skill 功能对需求的覆盖比例 | 4 |
| **维护度** | 最近更新时间 / commit 频率 | 3 |
| **安装量** | 社区使用验证度（>1000 装机量优先） | 3 |
| **文档** | 有无示例、参数说明、边界案例 | 3 |

### 1.3 判定
- **综合分 ≥ 15** 且无严重兼容问题 → **强烈推荐直接采用**，跳转第 4 步
- **综合分 10-14** → 列为候选但继续第 2 层漏斗
- **综合分 < 10 或无结果** → 记录为"Skills 生态无匹配"，继续第 2 层

---

## 第 2 步：第二层漏斗——PyPI / GitHub 开源库搜索

### 2.1 搜索渠道
同时使用以下三种方式：
- **PyPI 搜索**: `https://pypi.org/search/?q=<关键词>`
- **GitHub 搜索**: `https://github.com/search?q=<关键词>&type=repositories&l=Python`
- **Web 搜索补充**: 用 `web_search` 搜 "best python library for <核心动作>"

### 2.2 评估候选库
按以下硬门槛筛选，**任一不满足直接排除**：

| 门槛 | 标准 |
|---|---|
| **Star 数** | ≥ 500 |
| **最近 commit** | 近 6 个月内 |
| **Open Issues 比例** | < 20%（issues / (issues + closed)） |
| **License** | MIT / Apache 2.0 / BSD（项目涉及商用时） |
| **Python 版本兼容** | 支持 Python 3.10+（本项目基线） |
| **依赖深度** | 非纯 Python 的大型依赖（如 PyTorch、Selenium、Playwright）需特别说明 |

### 2.3 对候选库深度验证（至少抽 1 个做）
- 阅读 README 的 Quick Start 部分
- 检查 `pyproject.toml` / `setup.py` 看依赖清单
- 看 Issues 里最近的 Bug 报告严重程度
- 如果该库明显违反 `general-global-rule.md §3.1`（硬编码模型 ID 等），即使满足硬门槛也要排除

### 2.4 判定
- **找到 ≥ 1 个满足所有硬门槛的候选** → 继续第 4 步对比决策
- **全部不满足** → 记录为"开源生态无合格候选"，继续第 3 层

---

## 第 3 步：第三层漏斗——手写评估与 Skill 固化

### 3.1 确认手写的必要性
在宣布"只能手写"之前，再次自问：
1. 是否因为搜索关键词错了？换个词再搜一次
2. 是否需求描述太特殊？有没有方法拆成更通用的子问题，每个子问题用现成库解决
3. 是否是"粘合剂"类需求？这类通常就是要手写，确认后继续

### 3.2 手写方案评估
输出：
手写方案评估

预计代码量: <N 行>
主要风险: <如有>
测试策略: <如何验证>
是否可复用: <本项目独占 / 跨项目通用>


### 3.3 可复用判定 → 固化 Skill
如果 3.2 的"是否可复用"判定为"跨项目通用"，**主动向用户建议**：

> 💡 本次将手写的功能具有跨项目复用价值。建议开发完成后，用 `@skill-creator` 把它固化为一个新 Skill，未来其他项目可直接 `@find-skills <新 skill 名>` 调用。
> 
> 需要我在完成开发后启动 `@skill-creator` 流程吗？

用户同意则记入 TODO，验收后执行。

---

## 第 4 步：方案对比与推荐

无论前面三层漏斗走到哪一步，**统一输出**以下格式：
🔍 搜索漏斗结果
第 1 层（Skills 生态）

搜索关键词: [k1, k2, k3]
结果: <N 个候选 / 无匹配>
最优候选: <Skill 名> 综合分 X/20

第 2 层（开源库）

搜索关键词: [k1, k2, k3]
结果: <N 个候选 / 无合格候选>
最优候选: <库名> (GitHub X stars, 最近 commit YYYY-MM-DD)

第 3 层（手写评估）

是否必须手写: <是 / 否>
可复用性: <本项目独占 / 跨项目通用>
Skill 固化建议: <如有>

📊 方案对比表
方案类型开发成本长期维护综合推荐度方案 A: ...Skill低低⭐⭐⭐⭐⭐方案 B: ...开源库低中⭐⭐⭐⭐方案 C: 手写自研高高⭐⭐
✅ 推荐方案: <方案 X>
理由: <一句话核心理由>
❓ 是否采用此方案？

**然后停下等用户确认**，未确认不得进入编码。

---

## 反模式（禁止行为）

- ❌ 跳过第 0 步需求抽象，直接搜
- ❌ 只用 1 组关键词搜 1 次就说"找不到"
- ❌ 跳过第 1 层 Skills 生态直接搜 PyPI（Skills 是首选）
- ❌ 忽略硬门槛，推荐 stars < 100 或一年没更新的库
- ❌ 跨项目通用的手写需求不向用户建议固化 Skill
- ❌ 漏斗结果没整理就进入编码
- ❌ 搜索完不给方案对比表，只口头说"推荐 X"

---

## 与其他 Workflow 的关系

- 本 Workflow 通常由 `/plan-task` 的第 3 步调用
- 本 Workflow 的输出（推荐方案）会被 `/plan-task` 第 4 步纳入 TODO 清单
- 如最终选择"手写并固化 Skill"，`/verify-done` 完成后应触发 `@skill-creator`

---

## 快速模式（Trivial 任务跳过此 Workflow）

依照 `/plan-task` 的档位判定：
- **Trivial**（1-2 步骤、无新依赖）：可跳过本 Workflow，但 `/plan-task` 必须在计划中声明"本任务已判定为 Trivial，跳过搜索漏斗"
- **Normal 及以上**：强制走完本 Workflow
